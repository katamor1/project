# DOCXテンプレート設計規約

この規約は、IBM-BobのAIエージェントがJSONからWord設計書を生成するためのテンプレート作成ルールです。

## 1. プレースホルダ

単一値は以下の形式で記述します。

```text
{{document.document_name}}
{{$.document.version}}
{{json.target_system.system_name}}
```

3形式はいずれも同じJSON Pathとして扱われます。

## 2. 表の繰り返し

配列を表に展開する場合、表内に以下のマーカー行を置きます。

```text
[[TABLE:test_cases]]
```

その次の行をテンプレート行として扱います。

| ID | 観点 | 期待結果 |
|---|---|---|
| [[TABLE:test_cases]] | | |
| {{item.test_id}} | {{item.condition}} | {{item.expected_result}} |

生成後、マーカー行とテンプレート行は削除され、配列件数分の行が挿入されます。

## 3. 避けるべきテンプレート

- プレースホルダを複数のWordランに分割しない
- 1つのセルに複数の大きな配列を入れない
- ネストの深い巨大JSONを1表に押し込まない
- テキストボックス、SmartArt、図形内テキストにプレースホルダを置かない
- ヘッダー/フッター以外の特殊領域に重要情報を置かない

## 4. 推奨章立て

### 基本設計書

- 文書管理
- 目的・背景
- 対象範囲
- システムコンテキスト
- 機能要件
- 状態遷移/シーケンス
- データ設計/共有メモリ
- 外部インターフェース/API
- スレッド配置
- 非機能・安全・エラー
- 影響調査
- テスト受入条件
- 未確定事項

### 詳細設計書

- 文書管理
- トレーサビリティ
- 影響資産
- モジュール設計
- データ設計
- スレッド設計
- 状態遷移
- 関数詳細
- API/I/O
- 排他・整合性
- エラー・安全
- 性能
- build/test

### 機能テスト設計書

- 文書管理

- レビュー観点

## 5. 検証条件

生成後、以下がOKでなければ成果物にしません。

- DOCXがzipとして開ける
- `word/document.xml`がXMLとして読める
- 未置換`{{...}}`がない
- 未展開`[[TABLE:...]]`がない
- 本文が空でない
